## 已完成的后端能力

- 模型注册与动态加载
  - 支持 `paraformer`、`qwen3-asr`、`zipformer`、`whisper`、`funasr`
  - 模型配置从 `MODEL_REGISTRY_PATH` 指向的 JSON 文件读取
  - `engines/` 目录会自动发现 `*_engine.py` 文件并注册实现
- 系统健康检查
  - `GET /api/v1/system/health`
  - 返回系统状态、已加载模型、CPU、内存、GPU 指标
  - `psutil` 和 `pynvml` 缺失时自动降级
- 模型管理接口
  - `GET /api/v1/models`
  - `POST /api/v1/models/load`
  - `DELETE /api/v1/models/{model_name}`
  - `POST /api/v1/models/unload`
  - `GET /api/v1/models/health`
  - `POST /api/v1/models/refresh`
- 常规识别接口
  - `POST /api/v1/asr/transcribe`
  - `POST /api/v1/asr/batch`
- 批量任务接口
  - `POST /api/v1/asr/batch/tasks`
  - `GET /api/v1/asr/batch/tasks/{task_id}`
  - `GET /api/v1/asr/batch/tasks/{task_id}/result`
- WebSocket 流式识别
  - `WS /api/v1/asr/stream`
  - 支持开始消息、音频二进制分块、结束消息
  - 支持临时文本、最终文本、RTF、首字延迟等回传

## 模型配置方式

模型路径不写死在代码里，统一通过 `backend/config/model_registry.example.json` 这种 JSON 配置。

示例结构：

```json
{
  "models": {
    "whisper": {
      "engine": "whisper",
      "path": "D:/models/whisper",
      "device": "auto",
      "auto_load": false,
      "options": {}
    }
  }
}
```

你后续只需要把真实路径改到自己的配置文件里，并通过 `.env` 设置：

```bash
MODEL_REGISTRY_PATH=backend/config/model_registry.example.json
```

## 新增模型的接入方式

目标是尽量做到“新增模型文件即可接入系统”。当前约定如下：

1. 在 `backend/src/engines/` 新建一个 `xxx_engine.py`
2. 让该类继承 `BaseEngine`
3. 设置唯一的 `engine_name`
4. 实现 `_load_model()` 和 `_transcribe_impl()`
5. 在模型配置 JSON 里增加一个模型条目，`engine` 填对应的 `engine_name`

示例：

```python
from src.engines.base_engine import BaseEngine


class NewModelEngine(BaseEngine):
    engine_name = "new-model"

    def _load_model(self):
        return {"model_path": self.model_path}

    def _transcribe_impl(self, audio_path, language, hotwords, enable_itn):
        return f"transcript for {audio_path}"
```

因为 `engines/__init__.py` 会自动扫描 `*_engine.py`，所以不需要手工再去改注册表代码。

## 当前实现边界

- 目前 5 个模型实现仍是框架占位版本，已经具备统一加载和调用入口，但没有接真实推理库。
- 音频标准化目前先保留文件落盘和基础时长估算，真正的重采样、格式转换和 VAD 还可以继续往 `audio_processor.py` 里补。
- WebSocket 已经具备完整协议骨架，但当前是“基于完整转写结果模拟流式回传”，真实流式 ASR 接入后只需要替换服务层逻辑。
